## 项目定位

项目目标是把 ego 视角与 UMI 操作数据采集，从“临时脚本可运行”推进到“现场人员可稳定使用、新机器可复现部署、采集数据可直接进入训练链路”的工程系统。

系统围绕 Thor / Jetson 主机搭建，接入多路 GMSL2 相机与手持采集设备。手持设备同步提供 GMSL2 相机、六维力、触觉指尖、IMU 等信息，并通过 `tools/data_collection_gui` 提供采集控制、设备状态、数据集管理、episode replay 和 QC 能力。最终产物不是单个采集脚本，而是一套可部署、可自检、可维护的数据采集工作站。

## 核心工作

### 1. 采集链路从硬件到数据集打通

- 将多路 GMSL2 相机、手持采集设备、LeRobot v3 数据格式和前端 GUI 串成一条完整链路。
- 在 gateway 层统一暴露设备状态、采集控制、数据集根目录、replay / QC 等能力，减少前端直接理解硬件细节。
- 让采集输出显式标记手持设备数据来源，保证后续训练、回放和质量检查能区分相机观测、六维力、触觉与 IMU 数据。

### 2. 为 Thor / Jetson 实机环境补齐部署工程

现场部署的难点不在于“依赖能装上”，而在于新机每次都可能遇到系统库、网络、权限和设备地址问题。项目将这些问题沉淀成可复用部署清单：

- 梳理 JetPack 6 / Ubuntu 24.04 noble 下必须安装的 apt 运行时依赖。
- 处理手持采集设备运行库对旧版 `tinyxml2`、`urdfdom`、Boost soname 的兼容问题，并提供幂等的 compat symlink 脚本。
- 区分 GUI 最小运行依赖和 LeRobot 训练依赖，避免为了采集界面强行拉起完整训练环境。
- 针对国内网络环境，将 Node、npm、pip 的安装路径切换到可用镜像源，降低新机部署失败率。

### 3. 固化手持设备网络与启动方式

手持采集设备通过固定地址向 Thor 主机推送传感器 UDP 数据，Thor 重启后手动添加的网口地址会丢失。为解决这个问题，网络恢复逻辑被固化为 systemd 服务：

- 开机自动拉起 `enP2p1s0`。
- 幂等添加 `192.168.2.45/24`。
- 主动 probe 手持设备地址，刷新设备重上电后的 ARP / 邻居状态。
- 默认不修改内核反向路径过滤策略，需要时再通过环境变量显式打开。

同时整理了 gateway / Vite 的本地启动脚本模板，处理 ssh 退出后后台进程被带掉的问题，使用 `setsid + disown` 保证采集服务可以稳定脱离终端运行。

### 4. 建立上线前自检标准

为了让系统不是“部署者感觉能跑”，而是具备明确验收点，上线前自检被拆成几类：

- 手持采集设备 Python 绑定可以正常构造、启动和停止。
- gateway snapshot 能反映 Thor GMSL2 / 手持设备默认配置。
- 默认配置下相机与手持设备数量、设备详情和禁用逻辑符合预期。
- recorder 能根据配置选择 Thor 采集脚本，并兼容 legacy handheld 配置。
- 前端通过 Vite 暴露给开发机访问，现场人员可以直接在浏览器中操作采集。

## 系统结果

这套系统最终把 ego + UMI 数据采集沉淀成了一个可复制的工程交付：

- 新 Thor / Jetson 主机可以按清单完成部署，显著减少环境排错时间。
- 多相机、手持采集设备、GUI、gateway、LeRobot v3 数据集管理被放在同一条采集链路里。
- 数据采集后可以直接进入 replay、QC、训练前处理和后续策略训练流程。
- 部署文档记录了系统库兼容、国内网络、手持设备网口、后台启动等真实现场问题，便于后续扩机和交接。

这个项目的价值不只是接入多路相机和手持采集设备，而是把具身智能数据采集里最容易散落的部分：硬件、配置、GUI、数据格式、部署、自检和现场运维，收束成了一套能持续使用的基础设施。
